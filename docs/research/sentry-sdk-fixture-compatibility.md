# Sentry SDK Fixture Compatibility Gate

<!-- markdownlint-disable MD013 -->

Research date: 2026-05-25

## Purpose

The current architecture says Parallax should accept Sentry envelopes for error
ingestion. That claim is useful only if real Sentry SDKs can send useful data to
Parallax without surprising users.

This note tightens proof gate #5 from
[Strategic verdict and research coverage](strategic-verdict-and-research-coverage.md):

> Sentry envelope compatibility across real SDKs.

The decision: **do not call Parallax "Sentry-compatible" until a fixture suite
generated by current Sentry SDKs passes parser, normalization, grouping,
redaction, idempotency, and trace-correlation checks.**

## Current Primary-Source Checks

| Source | What matters for Parallax |
| --- | --- |
| [sentry-types envelope source](https://docs.rs/sentry-types/latest/src/sentry_types/protocol/envelope.rs.html) | The Rust protocol model treats an envelope as Sentry's ingestion format, with envelope headers such as `event_id`, `dsn`, `sdk`, `sent_at`, and `trace`, and item headers with item `type`, optional `length`, content type, filename, and attachment type. Current item variants include `event`, `session`, `transaction`, `attachment`, `check_in`, `log`, and `trace_metric`. |
| [sentry-python envelope source](https://getsentry.github.io/sentry-python/_modules/sentry_sdk/envelope.html) | The Python SDK documents the envelope constraints in code and specifically notes that each envelope may contain at most one `event` or `transaction`, not both. Its current source also emits profile/profile chunk items, proving item drift is real. |
| [sentry Rust crate 0.48.2](https://docs.rs/sentry/latest/sentry/) | The current Rust SDK integrates with Rust panics, backtraces, contexts, `anyhow`, `tracing`, OpenTelemetry, transports, and exposes the Sentry protocol/types. This is the first SDK fixture target because Parallax is Rust-first. |
| [sentry-tracing 0.48.2](https://docs.rs/sentry-tracing/latest/sentry_tracing/) | The tracing integration can map `tracing` events to breadcrumbs, events, logs, and spans. By default, high-severity events become Sentry events and ordinary events become breadcrumbs/spans. That directly affects which evidence arrives in an error envelope. |
| [Sentry issue grouping docs](https://docs.sentry.io/concepts/data-management/event-grouping/) | Sentry groups errors by considering `fingerprint` first, then stack trace, exception, and message. Stacktrace grouping depends heavily on in-app frames and normalized filenames/modules/context lines. Parallax should not promise exact grouping parity, but fixtures must preserve the material needed for deterministic Parallax grouping. |
| [Sentry fingerprint rules](https://docs.sentry.io/concepts/data-management/event-grouping/fingerprint-rules/) | Custom fingerprints can override or refine default grouping with `{{ default }}`. Parallax must preserve client-provided `fingerprint` and record whether it used it, ignored it, or combined it with its own grouping version. |
| [Sentry Relay repository](https://github.com/getsentry/relay) | Relay is the real Rust reference gateway, but processing mode performs full normalization/filtering/rate limiting and produces events into Kafka; its integration tests require Redis/Kafka. Parallax should use Relay as a compatibility reference, not copy its operational shape. |
| [Sentry distributed tracing SDK spec](https://develop.sentry.dev/sdk/telemetry/traces/distributed-tracing) | Sentry propagates `sentry-trace` and `baggage`, with optional W3C `traceparent` for OpenTelemetry interop. Parallax must map `contexts.trace` and propagated headers into OTLP trace joins. |

## Compatibility Levels

Use levels so the product language is honest:

| Level | Meaning | Product wording |
| --- | --- | --- |
| L0 parser | Accepts syntactically valid envelopes, handles unknown item types safely, and returns stable accept/reject behavior. | "Envelope endpoint." |
| L1 Rust error events | Current Rust SDK panic/error/message/tracing event envelopes normalize into Parallax `error_event` rows. | "Sentry-compatible Rust error ingestion." |
| L2 trace correlation | Rust SDK events carrying Sentry trace context join to OTLP spans/logs in bundles. | "Sentry errors linked to traces." |
| L3 grouping stability | Same fixture corpus produces stable Parallax fingerprints across parser releases. | "Deterministic Parallax grouping." |
| L4 multi-SDK smoke | Go and browser JavaScript error-event envelopes parse and normalize core fields. | "Sentry SDK-compatible error ingestion." |
| L5 broad parity | Multiple SDK languages, attachments, transactions, profiles/replay/session behavior, and server-side grouping parity. | Do not claim for MVP. |

The v0 target is **L1 + part of L2** for Rust. L4 is a later smoke gate. L5 is
not a Parallax goal unless the product deliberately expands into a Sentry
replacement.

## Fixture Generation Strategy

Fixtures should be generated, not hand-written.

```text
fixture app / sdk version / scenario
  -> Sentry SDK configured with Parallax fixture DSN or custom transport
  -> raw envelope bytes saved
  -> parser golden test
  -> normalized event snapshot
  -> grouping fingerprint snapshot
  -> redaction report snapshot
  -> optional evidence bundle snapshot
```

Every fixture directory should record:

- SDK name and version;
- language/runtime version;
- enabled features/integrations;
- scenario name;
- raw envelope hash;
- expected accepted/rejected items;
- expected normalized Parallax fields;
- expected redaction findings;
- expected grouping source.

Do not store live production envelopes as test fixtures. Use synthetic apps with
seeded PII/secret canaries from
[Redaction pipeline and secret safety](redaction-pipeline-and-secret-safety.md).

## Rust-First Fixture Matrix

| Scenario | SDK setup | Must prove |
| --- | --- | --- |
| `rust_panic_default` | `sentry` default integrations, panic hook enabled. | Event accepted; panic message/location/backtrace frames preserved; `debug_meta` captured when present; grouped from in-app frame or panic location. |
| `rust_capture_error_anyhow` | `sentry` + `anyhow` feature, nested context chain. | Error type/value/source chain preserved enough for agent summary and grouping fallback. |
| `rust_capture_message` | `capture_message` with level and tags. | Message-only event accepted; grouped by message fallback; low confidence recorded. |
| `rust_tracing_breadcrumbs` | `sentry-tracing` layer, debug/info breadcrumbs before captured error. | Breadcrumbs arrive bounded and redacted; structured fields land in expected context/tags. |
| `rust_tracing_event` | `tracing::error!` with fields and `tags.*`. | Sentry event item is emitted; Parallax preserves tags and context fields under policy. |
| `rust_explicit_fingerprint` | Client sets fingerprint. | Parallax records `fingerprint_source = client`; raw and normalized fingerprints are versioned/auditable. |
| `rust_trace_context` | Sentry trace context plus matching OTLP span/log fixture. | `trace_id`/`span_id` extracted; evidence bundle joins Sentry error to OTLP trace/log rows. |
| `rust_request_context` | HTTP context with query string, auth header, user id/email. | Safe route/method context kept; auth/cookie/user PII redacted or hashed; redaction report cites rules. |
| `rust_duplicate_event_id` | Same envelope sent twice. | Idempotent write; no duplicate issue event; duplicate marker or counter recorded. |
| `rust_unknown_item` | Event envelope plus unknown or unsupported item. | Event path still handled if valid; unsupported item dropped/ref-only with explicit outcome. |
| `rust_attachment_disabled` | Event plus attachment item. | Attachment rejected or metadata-only according to v0 policy; no raw attachment in agent bundle. |

This matrix is enough to validate the first product wedge without pretending to
support browser replay, profiling, release health, or full cross-language
grouping.

## Multi-SDK Smoke Later

After the Rust path is stable, add current-version smoke fixtures:

| SDK | Why | Minimum fixture |
| --- | --- | --- |
| Go | In scope by language filter and common for backend services. | Panic/error event with release/environment/tags and stack frames. |
| Browser JavaScript | Required for the frontend roadmap. | Minified frontend error with source-map metadata path recorded but not necessarily symbolicated in v0. |
| Node JavaScript | Common service runtime but outside the preferred systems-language filter. | Only if needed for Sentry migration demand; keep as compatibility smoke, not primary scope. |

Do not add broad language fixtures before the Rust parser and grouping tests are
stable. The adoption promise is "easy Sentry migration for the first target,"
not "full Sentry replacement."

## Parser And Oracle Decision

Parallax should not hand-roll the protocol from memory.

Use a two-layer approach:

1. **Production parser:** small Parallax-owned envelope parser or a carefully
   reviewed dependency on the MIT-licensed Sentry Rust protocol crates.
2. **Test oracle:** compare raw fixture parsing against current `sentry-types`
   behavior and, where useful, Relay schema snapshots.

Reasons:

- Sentry item names and item containers are evolving (`log`, `trace_metric`,
  profile chunks, and future item types).
- Relay's schema is the closest thing to the real ingestion contract.
- Parallax should keep unknown fields/items forward-compatible without turning
  unknown data into agent-visible context.

Any production dependency on Sentry protocol crates should be isolated behind a
Parallax `EnvelopeParser` trait so the gateway can replace it if the crate API or
license posture changes.

## Normalization Snapshot

For every accepted `event` item, the fixture expected output should assert:

```text
event_id
timestamp / received_at
platform
level
logger
release / dist / environment
transaction
exception.type / exception.value / mechanism
stack frames: function, raw_function, module, filename, abs_path, lineno, colno, in_app
threads fallback when exception stack is absent
request route/method and redacted headers/query
user fields under policy
tags and selected extra/context fields
contexts.trace trace_id/span_id/parent_span_id
breadcrumbs bounded + redacted
debug_meta/build image hints
fingerprint array
sdk name/version
```

The snapshot should also assert what Parallax intentionally discards. Silent
loss is the compatibility bug to avoid.

## Success And Failure Gates

Pass criteria for v0:

- current Rust SDK fixtures are generated from real SDK code, not handcrafted
  JSON;
- `event` item envelopes parse with and without optional item `length`;
- unsupported items do not poison supported `event` processing;
- malformed or oversized payloads fail before expensive normalization;
- normalized snapshots are stable and reviewable;
- grouping snapshots are versioned and deterministic;
- duplicate event IDs are idempotent;
- redaction fixtures leak zero seeded canaries into agent-visible JSON/Markdown;
- a trace-context fixture joins to an OTLP span/log row in the evidence bundle.

Failure criteria:

- the Rust SDK emits a common error envelope Parallax cannot parse;
- grouping snapshots churn for unchanged fixture input;
- unsupported items cause retries or total envelope loss when the event item was
  valid;
- trace IDs exist in the envelope but cannot be mapped into bundle joins;
- redaction requires dropping so much event context that the A1 bundle-value eval
  becomes weak.

## Product Implication

Sentry compatibility is a migration wedge, not a brand claim. The honest wording
for the first release should be:

> Compatible with Sentry Rust SDK error-event envelopes for panics, captured
> errors, messages, breadcrumbs, tags, release/environment, and trace context.

Avoid:

- "Drop-in Sentry replacement";
- "Sentry-compatible" without a supported SDK/version matrix;
- "supports Sentry SDKs" before Go/browser smoke fixtures pass;
- "same grouping as Sentry".

This keeps Parallax narrow, testable, and defensible.

## Bottom Line

Parallax should earn Sentry compatibility with a real SDK fixture suite. The
v0 milestone is not broad parity; it is current Rust SDK error envelopes parsed,
normalized, grouped, redacted, deduplicated, and joined into an evidence bundle.
Anything broader waits until the first wedge is proven.
